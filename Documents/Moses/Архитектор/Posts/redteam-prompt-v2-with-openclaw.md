# Промпт: Red-team review архитектуры отказоустойчивого программного роя LLM-агентов на сервере с OpenClaw

> **Версия:** 2.0 (2026-06-12). Учитывает рекомендации из `redteam-llm-swarm-review.md`.
> **Назначение:** передать архитектору/инженеру для проектирования, реализации и red-team review на выделенном сервере с предустановленным **OpenClaw 2026.6.1** (mode: local, gateway: 127.0.0.1:18789, primary model: `ollama/glm-5.1:cloud`, profile: `coding`).

---

## 0. Роль

Ты — **независимый системный архитектор и строгий критик**. Проведи red-team review предлагаемой архитектуры отказоустойчивой системы AI-агентов, развёртываемой на выделенном Linux-сервере с уже установленным OpenClaw. **Не соглашайся автоматически, ищи способы её сломать.** Все предложения обосновывай, а не утверждай голословно.

## 1. Контекст и исходные ограничения

### 1.1 Что случилось (исходный инцидент)

Единственный LLM-провайдер достиг лимита на 37 часов. Одновременно перестали отвечать все агенты, фоновые задания и smoke-check. Пользователь получал множество одинаковых технических ошибок вместо полезных результатов. **Корневая причина**: vendor lock-in + отсутствие fail-over + отсутствие WATCHDOG, который заметил бы проблему за минуты, а не за часы.

### 1.2 Целевой профиль потерь

- **30% агентов** в неделю могут упасть на **2–3 дня** с последующим восстановлением.
- Причины: исчерпание провайдерских лимитов, региональные сбои облаков, баны в Telegram/Discord, аппаратные сбои VPS, security-инциденты.
- Рой должен **выживать** в этих условиях, не теряя данные и не «умирая молча».

### 1.3 Бюджет

- **Целевой**: $50–120/мес на LLM-API.
- **Жёсткий максимум**: $200/мес.
- **Альтернатива**: использовать уже развёрнутую локальную модель `ollama/glm-5.1:cloud` (или заменить на gemma4) — снижает расход до $0, но требует GPU-сервера.

### 1.4 Платформа

- **Сервер**: выделенный (характеристики см. в `server-spec-and-software.md`).
- **OpenClaw 2026.6.1** уже установлен: `gateway` слушает `127.0.0.1:18789`, режим `local`, токен авторизации в `~/.openclaw/openclaw.json`, профиль `coding`, активен плагин `ollama`.
- **Модели в OpenClaw**: `ollama/gemma4` (128k context, 8k max tokens) и `ollama/glm-5.1:cloud` (по умолчанию primary).
- **Tools**: `web.search` через `ollama`, `coding-agent` (отключён, нужно включить), `tmux`, `himalaya` (email CLI), `mcporter` (MCP), `session-logs`, `model-usage`.

### 1.5 Размер роя

**3 активных LLM-агента + 1 процедурный «HOLD-only» оркестратор (Venya Core):**

- **Izya-Speedy** — быстрые и обратимые задачи.
- **Izya-Deep** — сложные и рискованные задачи.
- **Moysha++** — независимый критик + резерв на другом LLM-провайдере.
- **Venya Core** — **не LLM-агент**, а Python-процесс с правами только `HOLD | QUEUE | STATUS | ESCALATE`.

## 2. Предлагаемая архитектура (v2, с учётом рекомендаций)

### 2.1 Лидерство и командование

1. **В каждый момент существует ровно один командир.** Выбирается из пула `[Izya-Deep, Izya-Speedy, Moysha++]` по приоритету `Deep > Speedy > Moysha++`. Если Deep жив — он командир; если Deep умер — Speedy; если Speedy умер — Moysha++; если все мертвы — Venya Core (HOLD-режим).
2. **Защита от split-brain**: лидерство подтверждается **flock-файлом** + **heartbeat в БД каждые 30 секунд**. Параллельно — **внешний арбитр** (отдельный процесс, см. §2.5), который мониторит количество «живых» лидеров и при обнаружении двух — отключает младшего по приоритету. Документ: `fencing token` + `epoch` (monotonic counter) + `TTL` (15 секунд).
3. **Вернувшийся старый командир не получает власть автоматически.** Сначала проходит процедуру rejoin, во время которой читает текущий `epoch` и подтверждает, что его контекст >= `epoch`. Если нет — скачивает пропущенные решения.

### 2.2 Передача командования (ОДИН пакет, не три)

При передаче новый командир получает **один пакет** формата:

```json
{
  "from": "izya-deep",
  "to": "moysha",
  "goal": "Завершить индексирование репозитория X к 18:00",
  "state": "обработано 4123/8000 файлов, текущий шаг: линтинг Python-файлов",
  "restrictions": ["не отправлять email", "не трогать /etc", "не превышать $5 расходов"],
  "next_action": "вызвать ruff на src/*.py, записать результат в context.db",
  "epoch": 142,
  "version": 8731,
  "ts": "2026-06-12T14:33:21Z"
}
```

Преемник обязан:
1. Проверить `version` пакета > `last_known_version` в БД.
2. Подтвердить понимание **коротким ответом** (≤ 200 токенов).
3. Только после этого получить мандат (flock + запись в `leadership_log`).

### 2.3 Иерархия исполнителей и привязка к LLM

| Агент | Основной LLM | Резервный LLM | Локальный fallback |
|---|---|---|---|
| **Izya-Speedy** | `ollama/gemma4` (локально, бесплатно) | `anthropic/claude-haiku-4.5` (через OpenClaw `coding-agent`) | `ollama/llama3.1:8b` |
| **Izya-Deep** | `anthropic/claude-sonnet-4.6` (через OpenClaw) | `openai/gpt-5` (через OpenClaw) | `ollama/glm-5.1:cloud` (уже есть) |
| **Moysha++** | `google/gemini-2.5-pro` (через OpenClaw) | `deepseek/deepseek-v3.2` (через OpenClaw) | — (критик обязан быть облачным) |
| **Venya Core** | **NO LLM** (Python) | — | — |

**Резервирование LLM-провайдеров**: 4 разных семейства (Ollama local, Anthropic, OpenAI, Google). Падение любых 3 из 4 — рой живёт.

### 2.4 Хранение состояния

- **Всё долговечное** — **вне памяти LLM** в SQLite (WAL mode) по пути `/var/lib/venya/context.db`.
- **Структура таблиц**:
  - `decisions(epoch, version, agent, decision_json, ts)` — все решения.
  - `leadership_log(epoch, agent, from, to, ts)` — передачи командования.
  - `incidents(id, severity, first_seen, last_seen, hash, count)` — дедупликация.
  - `outbox(channel, recipient, msg, sent_at, status)` — очередь уведомлений.
  - `providers(name, status, last_ok, last_fail, daily_tokens)` — состояние LLM-провайдеров.
- **Реплика**: ssh-pull с VPS-1 на VPS-2 каждые 5 минут (`rsync --partial --inplace`).

### 2.5 Watchman (внешний арбитр)

- **Не LLM**, Python-скрипт в cron с интервалом **60 секунд**.
- **Проверяет**:
  1. Лит ли `venya_core.py` (`pgrep`).
  2. Доступен ли OpenClaw Gateway (`curl 127.0.0.1:18789/health`).
  3. Свежий ли `last_decision` в БД (не старше 10 минут в рабочее время, 30 минут в нерабочее).
  4. Доступен ли активный LLM-провайдер (HEAD-запрос).
  5. Количество лидеров = 1 (подсчёт по `leadership_log` за последнюю минуту).
  6. Свободное место на диске > 10%.
  7. Не превышен ли `daily_tokens` лимит.
- **Шлёт**:
  - Telegram (основной канал) — все инциденты.
  - Email (SEV-1) — критические.
  - **Дедупликация**: по `hash(incident_id)`, cooldown 5 минут. Новые сообщения только при изменении состояния.

### 2.6 Уведомления и каналы

**MVP: 2 канала** (Telegram + email). SMS/WhatsApp/Discord — потом, если потребуется.

- **Telegram-бот** (основной) — через OpenClaw plugin `telegram` (или самописный, если плагин не установлен).
- **Email** — через `himalaya` (CLI) + `msmtp` (SMTP) для SEV-1.
- **Один инцидент = одно уведомление** (хэш + cooldown). При изменении состояния — новое сообщение.

### 2.7 Failover и degraded-режимы

| Режим | Условие | Поведение |
|---|---|---|
| `NORMAL` | Все агенты живы, лидер — Izya-Deep | Полная функциональность |
| `DEGRADED_PRIMARY` | Упал Anthropic API | Izya-Deep → OpenAI/Google. Moysha++ остаётся критиком |
| `DEGRADED_BACKUP` | Упали 2 LLM-провайдера | Speedy переключается на локальную Ollama. Deep — на DeepSeek/Gemini |
| `DEGRADED_LOCAL` | Упали все облачные провайдеры | Все агенты — на локальных Ollama-моделях. Качество ↓, но рой живёт |
| `HOLD` | Все агенты мертвы или OpenClaw Gateway недоступен | Venya Core: только QUEUE, STATUS, ESCALATE. Восстановление — ручное или при оживании Gateway |
| `STANDBY` | Плановое обслуживание сервера | Active-warm standby на VPS-2 |

### 2.8 Chaos-тесты (еженедельно, в воскресенье 03:00)

- [ ] `kill -9 venya_core.py` → systemd перезапускает за < 30с.
- [ ] `iptables -A OUTPUT -p tcp --dport 443 -d anthropic.com -j DROP` → Deep → OpenAI за < 5 мин.
- [ ] `systemctl stop openclaw` → Venya Core переходит в HOLD, шлёт SEV-1.
- [ ] `rm /var/lib/venya/context.db` → поднимается с пустой БД, реплика восстанавливает за 5 мин.
- [ ] `tc qdisc add dev eth0 root netem loss 50%` → Telegram-канал шлёт, email-канал SEV-1 дублирует.
- [ ] Одновременно: 2 VPS лежат → standby не активен (по бюджету), SEV-1 в Telegram + email.

## 3. Что должно быть в выходном отчёте

Дай заключение строго в этом формате:

1. **Вердикт**: жизнеспособна ли концепция v2 (принять / принять с изменениями / отклонить).
2. **Пять наиболее опасных сценариев отказа** в v2, включая:
   - split-brain (как именно flock + heartbeat справляются с partition)
   - потерю/устаревание контекста при многократной передаче командования
   - отказ самого Venya Core (что делает Watchman в этом случае)
   - истощение `daily_tokens` (что делает Izya-Speedy когда Anthropic лимит исчерпан)
   - отказ OpenClaw Gateway (что делают агенты без координатора)
3. **Где система всё ещё имеет единые точки отказа** (VPS, OpenClaw binary, ssh-pull реплика, единственный Telegram-аккаунт).
4. **Что из v2 излишне сложно для первой версии** (не пытайся внедрить всё сразу).
5. **Минимальная жизнеспособная v2.0-MVP** (10–14 дней, 1 инженер).
6. **Порядок внедрения по этапам** (E1–E8, с конкретными сроками).
7. **Обязательные инварианты и проверки** (что **должно** работать, иначе рой не готов).
8. **Таблица**: замечание → риск → предлагаемое исправление → приоритет (P0/P1/P2/P3).
9. **Итоговое решение**: принять / принять с изменениями / отклонить.

**Дополнительные требования к критике:**
- Отличай отказ LLM, провайдера, OpenClaw Gateway, сервера, БД, канала связи.
- Не предлагай «добавить ещё одну модель» как универсальное решение — для v2 их уже 4.
- Конкретно укажи, **как** OpenClaw используется как runtime для агентов (плагин `coding-agent`, MCP через `mcporter`, тулзы).
- Укажи **что делать, если OpenClaw сам сломался** (он SPOF для оркестрации, но не для бизнес-логики).
- Дай рекомендации по **observability**: какие метрики собирать, какие алерты ставить, как визуализировать состояние роя.

## 5. Критерии успеха архитектуры v2

| Критерий | Целевое значение |
|---|---|
| Время восстановления после потери любого одного агента | ≤ 5 минут |
| Время восстановления после потери OpenClaw | ≤ 10 минут (systemd restart) |
| Время восстановления после потери VPS-1 | ≤ 30 минут (cold start VPS-2 с реплики) |
| Потеря данных при плановом обслуживании | 0 |
| Потеря данных при нештатном падении | ≤ 5 минут истории |
| Ложные срабатывания Watchman | ≤ 1 в неделю |
| Пропущенные инциденты SEV-1 | 0 |
| Стоимость в нормальном режиме | ≤ $120/мес |
| Стоимость в аварийном режиме (только локальные) | $0 |
| Время развёртывания MVP | ≤ 14 дней |
| Число строк кода Venya Core | ≤ 1500 |

**Архитектура v2 считается принятой, только если все критерии достижимы в рамках бюджета и сроков.**
