# Глава 6. Алгоритмы глубже: PSO, ACO, BFT

Три семейства алгоритмов, без которых невозможно понять роевые системы и их устойчивость:
- **PSO** — Particle Swarm Optimization. Метаэвристика для оптимизации.
- **ACO** — Ant Colony Optimization. Метаэвристика на основе феромонов.
- **BFT** — Byzantine Fault Tolerance. Устойчивость к злонамеренным/сломанным узлам.

---

## 1. Particle Swarm Optimization (PSO)

### Идея

Группа «частиц» летает в пространстве решений. Каждая частица помнит:
- свою лучшую позицию (где она лично была лучшей по fitness)
- глобально лучшую позицию (где кто-то из роя был лучшим)

На каждом шаге частица обновляет скорость, комбинируя:
- инерцию (продолжать лететь туда, куда летела)
- когнитивный компонент (тянуться к своей лучшей позиции)
- социальный компонент (тянуться к лучшей позиции роя)

Из этих трёх сил рождается движение к оптимуму.

### Математика

Для частицы *i* в момент *t*:

```
v_i(t+1) = w·v_i(t) + c1·r1·(pbest_i - x_i(t)) + c2·r2·(gbest - x_i(t))
x_i(t+1) = x_i(t) + v_i(t+1)
```

Где:
- `v_i` — вектор скорости частицы *i*
- `x_i` — вектор позиции (кандидат решения)
- `w` — коэффициент инерции (обычно 0.7–0.9)
- `c1` — когнитивный коэффициент (обычно 1.5–2.0)
- `c2` — социальный коэффициент (обычно 1.5–2.0)
- `r1, r2` — случайные числа в [0, 1]
- `pbest_i` — лучшая личная позиция частицы
- `gbest` — лучшая позиция всего роя

### Варианты

- **GBest PSO** — все тянутся к одному глобальному лучшему. Быстрее сходится, но попадает в локальные минимумы.
- **LBest PSO** — каждый тяготеет к лучшему в своём «кольце» соседей. Медленнее, но надёжнее.
- **Multi-Objective PSO** — несколько целевых функций, фронт Парето.
- **Binary PSO** — для дискретных задач.
- **Quantum PSO** — частицы в квантовом пространстве (для исследований).
- **Adaptive PSO** — `w`, `c1`, `c2` подстраиваются по ходу.

### Применения

- Обучение нейросетей (веса, гиперпараметры)
- Планирование траекторий дронов
- Расчёт энергосистем
- Финансовая оптимизация
- Компоновка антенн (NASA, 2006 — оптимизация антенны для ST5)

### Сильные стороны

- Прост в реализации
- Мало параметров
- Хорошо работает на непрерывных задачах
- Параллелится естественно

### Слабые стороны

- Попадает в локальные оптимумы
- Не работает с дискретными переменными (без модификаций)
- Не гарантирует глобальный оптимум
- Скорость сходимости зависит от `w`, `c1`, `c2`

### Интуиция

Представьте стаю птиц, ищущую место с лучшей едой. Каждая птица помнит, где лично ей было лучше всего, и слышит от других, где им было лучше. Итог: вся стая движется к глобально лучшему месту, но не забывает исследовать.

## 2. Ant Colony Optimization (ACO)

### Идея

Искусственные муравьи ходят по графу, оставляя «феромоны» на хороших путях. Чем короче путь, тем больше феромонов на единицу длины. Другие муравьи с большей вероятностью выбирают пути с феромонами. Плохие пути со временем «испаряются».

Из этого возникает кратчайший путь.

### Математика (для задачи коммивояжёра, TSP)

Городов N, муравьёв M.

**Вероятность выбора следующего города k из текущего i:**

```
p_ik = (τ_ik^α · η_ik^β) / Σ_j (τ_ij^α · η_ij^β)
```

Где:
- `τ_ik` — количество феромонов на ребре (i, k)
- `η_ik = 1/d_ik` — видимость (inverse расстояния)
- `α` — вес феромонов (обычно 1)
- `β` — вес видимости (обычно 2–5)
- Сумма в знаменателе — по всем ещё не посещённым городам

**Обновление феромонов (после того как все муравьи прошли маршрут):**

```
τ_ik(t+1) = (1 - ρ)·τ_ik(t) + Σ_m Δτ_ik^m
```

Где:
- `ρ` — коэффициент испарения (обычно 0.1–0.5)
- `Δτ_ik^m = Q / L_m` — если муравей m прошёл по ребру (i, k), где `L_m` — длина его маршрута, `Q` — константа
- Иначе `Δτ_ik^m = 0`

### Варианты

- **Ant System (AS)** — оригинальный, Dorigo 1992
- **Ant Colony System (ACS)** — локальное обновление феромонов, элитизм
- **MAX-MIN Ant System (MMAS)** — ограничение min/max феромонов
- **Ant Colony Optimization with Multiple Colonies** — несколько колоний с разными целями (multi-objective)
- **Pareto ACO** — для многокритериальных задач

### Применения

- Задача коммивояжёра и обобщения (VRP)
- Маршрутизация в телекоммуникационных сетях
- Планирование производства (job-shop scheduling)
- Раскройка материалов
- Биоинформатика (свёртывание белков)
- Маршрутизация в mesh-сетях IoT (глава 3)
- Распределение задач между роботами

### Сильные стороны

- Параллелится естественно (муравьи не зависят друг от друга в одном шаге)
- Хорошо работает на дискретных задачах
- Адаптивен — может работать в динамически меняющейся среде
- Находит несколько хороших решений, а не одно

### Слабые стороны

- Медленная сходимость
- Много параметров (α, β, ρ, Q, M, начальные феромоны)
- Может «застрять» в локальном оптимуме, если феромонов слишком много
- Не работает с непрерывными переменными (без модификаций)

### Связь с реальной природой

Настоящие муравьи (Linepithema humile и др.) действительно используют феромоны для маршрутизации. Эксперименты Госса, Дориго и др. в 1989 году показали, что колония муравьёв находит кратчайший путь между едой и гнездом, даже если особи слепы.

Ключевое отличие от реальной природы: реальные муравьи используют **stigmergy** (изменение среды), а не явное общение. Это вдохновило на исследования **stigmergic coordination** в роевых системах в целом.

## 3. Byzantine Fault Tolerance (BFT)

### Проблема

**Византийская проблема** (Lamport, Shostak, Pease, 1982):

Несколько генералов окружили город. Они могут общаться только через посланников. Некоторые генералы — предатели. Все лояльные генералы должны прийти к **единому решению** (атаковать или отступить), иначе проиграют. Предатели могут посылать разные сообщения разным генералам.

В распределённой системе:
- **Генералы** = узлы
- **Посланники** = сеть
- **Предатели** = узлы, которые дают сбой или злонамеренны
- **Единое решение** = консенсус

**Byzantine fault** — это **произвольный** сбой. Узел может:
- молчать,
- посылать разные сообщения разным узлам,
- координироваться с другими сломанными узлами.

Это самая сложная модель отказа. Более простые:
- **Crash fault** — узел просто молчит
- **Omission fault** — узел пропускает сообщения
- **Timing fault** — узел не успевает вовремя

### Теорема

**Невозможно** достичь консенсуса с **N** узлами, если **f** предателей, при **N ≤ 3f**.

То есть:
- 3 узла, 1 предатель — **невозможно** договориться
- 4 узла, 1 предатель — возможно
- 7 узлов, 2 предателя — возможно
- 10 узлов, 3 предателя — возможно
- В общем: `N ≥ 3f + 1`

Это фундаментальный предел. Доказано в 1982 году.

### Практические алгоритмы

#### PBFT (Practical Byzantine Fault Tolerance, 1999)

**Главный алгоритм** для BFT-консенсуса. Castro и Liskov, MIT.

**Три фазы:**
1. **Pre-prepare** — лидер предлагает порядок запросов
2. **Prepare** — узлы подтверждают, что видели предложение лидера
3. **Commit** — узлы подтверждают, что видели достаточно prepare-сообщений

**Требования:**
- `N ≥ 3f + 1`
- `O(N²)` сообщений на запрос
- Работает при асинхронной сети (в отличие от Paxos/Raft, которые требуют синхронности)

**Применения:**
- Hyperledger Fabric (блокчейн IBM)
- Zilliqa (блокчейн с шардингом)
- Другие enterprise-блокчейны

#### Проблемы PBFT

- **O(N²)** — плохо масштабируется
- **Лидер — узкое место** (хоть и заменяется)
- **Сложность реализации**

#### Решения для масштабирования

- **HotStuff** (Yale, 2018) — линеаризует BFT, O(N) сообщений, используется в Facebook Libra/Diem
- **Tendermint** — для блокчейна Cosmos
- **Algorand** — VRF для выбора лидера, масштабируется до тысяч узлов
- **Solana** — оптимистичный BFT, тысячи узлов

### BFT vs. Crash Fault Tolerance

| Свойство | Crash FT (Raft, Paxos) | Byzantine FT (PBFT) |
|---|---|---|
| Устойчивость к молчанию | Да | Да |
| Устойчивость к лжи | Нет | Да |
| Требование к N | `N ≥ 2f + 1` | `N ≥ 3f + 1` |
| Сообщений на запрос | O(N) | O(N²) или O(N) для HotStuff |
| Сложность | Низкая | Высокая |
| Где применяют | etcd, Consul, Kafka | Блокчейн, критические системы |

### BFT в роевых системах

Зачем BFT в рое? Несколько причин:

1. **Атаки:** военный или критический рой может подвергаться атакам. Часть узлов может быть взломана.
2. **Сбои:** произвольный сбой в оборудовании может дать произвольные данные.
3. **Multi-organization:** если рой принадлежит нескольким организациям, не все доверяют друг другу.

**Примеры:**
- **Drone swarms с BFT** — военные (Perdix, LOCUST) используют Byzantine-устойчивые протоколы
- **Блокчейн-валидаторы** — стандартное применение
- **Multi-robot SLAM** — каждый робот строит карту, нужна устойчивость к сломанным датчикам
- **Swarm consensus** — если часть роя отравлена плохими данными

**Реальные протоколы:**
- **DRBFT (Dynamic Round-based BFT)** — для роевых сетей
- **BFT-SMaRt** — Java-библиотека
- **Tendermint** — для блокчейна, может использоваться в IoT-рое

### Когда НЕ нужен BFT

- Закрытая корпоративная сеть, где все узлы доверенные
- Один владелец, физический контроль над оборудованием
- Узлы дешёвые, можно просто игнорировать «сумасшедшие» данные

В этих случаях достаточно **crash fault tolerance** (Raft, Paxos) или вообще без формального консенсуса (gossip, eventual consistency).

## Сравнение трёх алгоритмов

| | PSO | ACO | BFT |
|---|---|---|---|
| **Цель** | Оптимизация | Оптимизация | Консенсус |
| **Тип** | Метаэвристика | Метаэвристика | Протокол консенсуса |
| **Где применяют** | ML, планирование | Маршрутизация, scheduling | Блокчейн, военные рои |
| **Гарантии** | Нет (эвристика) | Нет (эвристика) | Да (формальные) |
| **Устойчивость к сбоям** | Средняя (выбросы) | Средняя (выбросы) | Высокая (3f+1) |
| **Масштабируемость** | Высокая (тысячи частиц) | Средняя | Низкая (PBFT), средняя (HotStuff) |

## Комбинации

В современных системах часто комбинируют:
- **PSO + BFT** — PSO для оптимизации параметров роя, BFT для безопасного обмена данными
- **ACO + mesh** — феромоны хранятся в распределённой mesh-сети
- **PSO + LLM** — современные исследования, LLM-агенты используют PSO для коллективного принятия решений
- **ACO + ACO** — несколько колоний с разными феромонами решают multi-objective задачи
