# Глава 4. Программные агенты и LLM-рой

## Что такое программный рой

Группа автономных программ, которые координируются для общей задачи. От обычного микросервиса отличается:
- агент сам принимает решения (не просто исполняет workflow),
- агенты общаются друг с другом (не только через шину данных),
- нет жёсткого контракта API — есть намерения и контекст,
- рождаются, умирают и реплицируются по задачам.

Исторически — классические мультиагентные системы (MAS) с 1990-х. С 2023 года — взрывной рост благодаря LLM, потому что LLM-агенту можно дать инструкцию на естественном языке, и он сам поймёт, что делать.

## Эволюция подходов

### 1. Классический MAS (1990–2015)
- Агент = Belief-Desire-Intention (BDI) архитектура
- Логика, планировщики (STRIPS, PDDL), дедуктивные правила
- Применение: автоматизация, складское планирование, военные моделирования
- **Плюсы:** формальная верификация, предсказуемость
- **Минусы:** дорого в разработке, узкая специализация

### 2. Эра фреймворков (2016–2022)
- **JADE** (Java, с 2000, но популярен до сих пор) — FIPA-совместимый, ACL-сообщения
- **SPADE** (Python) — агенты как акторы
- **Ray** (UC Berkeley) — actor model, scale to thousands
- **Celery + Redis** — task queue, не совсем агенты, но близко
- **ROS (Robot Operating System)** — для роботов, но архитектура узлов/pub-sub вдохновила
- **Паттерн:** Master-Worker, Pub-Sub, Saga

### 3. LLM-агенты (2023–н.в.)
- Агент = LLM + системный промпт + инструменты + память
- Общение — на естественном языке (или структурированном JSON)
- Инструменты — вызов функций (function calling), код, API, базы данных
- Память — кратковременная (контекст LLM) и долговременная (vector DB)

## Архитектуры LLM-роёв

### Иерархия (master/worker)
Один «главный» LLM декомпозирует задачу и распределяет по «рабочим» агентам.

**Примеры:**
- **AutoGPT** (2023) — главный агент ставит себе задачи и исполняет
- **BabyAGI** (2023) — главный агент создаёт список задач, исполнители выполняют
- **CrewAI** (2023–н.в.) — главный + команда, явные роли
- **LangGraph** (LangChain) — графовая оркестрация, явные роли

**Плюсы:** просто понять, легко отлаживать, лидер — единая точка оптимизации.
**Минусы:** лидер — узкое место, лидер может галлюцинировать.

### Децентрализованный обмен сообщениями
Агенты общаются peer-to-peer, нет центрального координатора.

**Примеры:**
- **ChatDev** (2023) — агенты в виртуальной компании, общаются по протоколу
- **MetaGPT** (2023) — агенты в симуляции стартапа
- **Multi-Agent Debate** (Du et al., 2023) — несколько LLM спорят, судья выбирает лучший ответ

**Плюсы:** устойчивость к отказу одного агента, разнообразие перспектив.
**Минусы:** расход токенов, сложнее контролировать, может «зациклиться».

### Роевое поведение
Агенты следуют простым локальным правилам, глобальное поведение возникает.

**Примеры:**
- **AgentVerse** (2023) — агенты в симуляции, формируют группы под задачи
- **EconAgent** (2024) — экономические агенты с собственными интересами
- **Generative Agents** (Stanford, 2023) — симуляция деревни из 25 LLM-агентов

**Плюсы:** масштабируется, не нужно проектировать протоколы.
**Минусы:** трудно предсказать поведение, может возникнуть «стадный эффект».

### Blackboard pattern
Общее хранилище (blackboard), агенты читают и пишут в него.

**Примеры:**
- **LangChain Multi-Agent** (старые версии)
- **LLM-Modulo** (2024) — несколько агентов предлагают решения, verifier отвергает плохие
- Применение: поиск уязвимостей в коде, где агенты пишут эксплойты, другие проверяют

**Плюсы:** естественная координация, агенты могут не знать друг о друге.
**Минусы:** shared state — узкое место, race conditions.

## Архитектурные паттерны LLM-роёв

### ReAct (Reason + Act)
Агент думает → выбирает инструмент → вызывает → наблюдает результат → повторяет. Базовый паттерн с 2022.

### Plan-and-Execute
Главный агент составляет план (часто в виде графа), затем исполнители выполняют шаги. План можно пересмотреть.

### Reflexion
Агент рефлексирует над ошибками, запоминает, что пошло не так. Самосовершенствование.

### Tree of Thoughts
Несколько «мыслей» параллельно, оценка, выбор лучшей ветки. Дороже, но точнее на сложных задачах.

### Mixture of Agents
Несколько LLM отвечают на вопрос, аггрегатор выбирает лучший ответ или синтезирует.

### Agentic RAG
Агент сам решает, какие документы искать, какие запросы делать, когда остановиться.

### Toolformer-style
Агент учится вызывать инструменты в процессе дообучения. Стандартизировано в OpenAI Function Calling, Anthropic Tool Use.

## Кто «главный» в LLM-рое

Зависит от архитектуры:
- **CrewAI, BabyAGI, AutoGPT** — явный мастер
- **ChatDev, MetaGPT** — нет мастера, есть SOP (стандартная операционная процедура)
- **Generative Agents** — нет мастера, агенты взаимодействуют на основе памяти и контекста
- **Multi-Agent Debate** — нет мастера, есть «арбитр» (судья)
- **AgentVerse** — нет мастера, агенты самоорганизуются в группы

## Что если агент «погиб»

В программном рое «гибель» = краш, бесконечный цикл, или ответ не приходит вовремя:

- **Master/worker:** переизбрание мастера или просто перезапуск worker-а
- **Decentralized:** система замечает по таймауту, убирает «труп» из списка
- **LLM-специфика:** если агент галлюцинирует — он формально «жив», но опасен. Поэтому важны:
  - **rate limiting** (лимит вызовов LLM),
  - **timeouts** на каждое действие,
  - **verifier-агенты** (отдельный агент проверяет результат),
  - **constitutional AI** (критерии, по которым результат отвергается).

## Проблемы LLM-роёв

### Стоимость
- 10 агентов × 100 вызовов каждый = 1000 LLM-вызовов
- При GPT-4 уровня: $30–$300 за сложную задачу
- Оптимизации: маленькие модели (Haiku, локальные), кэширование, переиспользование контекста

### Ненадёжность
- LLM может вернуть JSON с пропущенным полем
- Может зациклиться
- Может «забыть» контекст в длинном диалоге
- Может отказаться от задачи «по соображениям этики»

### Безопасность
- **Prompt injection:** злоумышленник в данных, которые агент читает, может перехватить управление
- **Tool misuse:** агент вызывает API, которое не должен
- **Excessive agency:** агент делает то, чего не должен (например, удаляет базу)
- **Решения:** sandboxing, явные allow-list инструментов, валидация выходов, аудит

### Координация
- Когда агенты общаются на естественном языке, они могут не понять друг друга
- Каждый вызов — это «переводчик», шанс ошибки
- Решение: structured outputs (JSON schema), явные протоколы (FIPA-ACL, KQML — старые, но идеи живые)

### Эмерджентное поведение
- Хорошо: агенты находят неожиданные решения
- Плохо: агенты образуют «петли обратной связи» — A вызывает B, B вызывает A
- Решение: timeouts, лимит на итерации, мониторинг
