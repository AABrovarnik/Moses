# Глава 9. Безопасность роя

## Почему безопасность роя — отдельная большая тема

Рой — это **распределённая система с физическим миром**. Атака на рой может привести к:
- физическому ущербу (дрон упал на человека)
- компрометации данных (видео с камер)
- отказу миссии (спасательная операция провалена)
- политическому скандалу (дроны-шпионы)
- массовому ущербу (500 дронов одновременно выведены из строя)

Сложность:
- **Большая поверхность атаки** — каждый агент = точка входа
- **Mesh-сеть** — компрометация одного узла = компрометация всей сети
- **Доверие между агентами** — если агент B доверяет агенту A, то компрометация A = компрометация B
- **Real-time ограничения** — некогда «обновлять антивирус»
- **Эмерджентное поведение** (глава 8) — атака может вызвать непредсказуемые последствия

## Модель угроз

### STRIDE — классификация угроз

| Категория | Угроза | Пример в рое |
|---|---|---|
| **S**poofing | Подмена личности | Фальшивый агент выдаёт себя за лидера |
| **T**ampering | Изменение данных | Агент передаёт ложные координаты |
| **R**epudiation | Отказ от действия | Агент делает что-то, потом отрицает |
| **I**nformation disclosure | Утечка | Видео с камер дронов утекает |
| **D**enial of service | Отказ в обслуживании | Атака на mesh, рой не может общаться |
| **E**levation of privilege | Повышение привилегий | Агент берёт на себя роль лидера |

### Атаки на разные слои

| Слой | Типичные атаки | Защита |
|---|---|---|
| **Физический** | Подмена/уничтожение агента | Геофенсинг, видеонаблюдение |
| **Связь** | Перехват (sniffing), jamming, MITM | Шифрование, frequency hopping |
| **Сеть** | DDoS, ARP spoofing, routing attacks | Mesh-протоколы с аутентификацией |
| **Протокол консенсуса** | Sybil, eclipse, partition | BFT, проверка подписей |
| **Приложение** | Prompt injection, code injection | Sandboxing, валидация входов |
| **LLM-агент** | Jailbreak, tool misuse | Constitutional AI, allow-list tools |
| **Данные** | Подмена карты, координат | Криптографические подписи |
| **Алгоритм** | Adversarial examples (ML) | Adversarial training, проверка выходов |

## Реальные инциденты

### Mirai botnet (2016)
- 600 000 IoT-устройств заражены трояном
- Использованы для DDoS-атаки на Dyn DNS
- Dyn обслуживал Twitter, Reddit, GitHub, Netflix
- Лежало 2/3 интернета в восточном побережье США
- **Урок:** дешёвые IoT-устройства без шифрования = оружие массового поражения

### Stuxnet (2010)
- Целевая атака на иранские центрифуги для обогащения урана
- Использовал 4 zero-day уязвимости Windows
- **Урок:** даже изолированные системы можно атаковать через USB и заражённое оборудование

### Hack the Drone (2015, DEF CON)
- Конференция по безопасности, демо-взлом потребительских дронов
- Взлом DJI, Parrot, 3DR
- **Урок:** потребительские дроны небезопасны by design

### Iranian capture of US RQ-170 Sentinel (2011)
- Секретный разведывательный дрон
- Иран якобы перехватил GPS-сигнал и посадил дрон
- **Споры:** была ли это реальная атака или пропаганда, но опасения обоснованы
- **Урок:** GPS-spoofing реально

### Петербургский GPS-spoofing (2017)
- Корабли в порту Петербурга показывали позицию в аэропорту
- Возможно — работа российской системы ГЛОНАСС-глушения или тест
- **Урок:** GPS-spoofing работает в масштабе

### DJI Drone Audit (2017+)
- Министерство внутренней безопасности США обвинило DJI в отправке данных на серверы в Китае
- DJI — крупнейший производитель дронов
- Армия США запретила DJI в 2017
- **Урок:** supply chain attacks реальны

### Verkada Camera Hack (2021)
- Стартап Verkada (IoT-камеры) был взломан
- 150 000 камер, в том числе в Tesla, Equinox, полиции
- Атакующие получили доступ к видео
- **Урок:** IoT-платформы = single point of failure

## Типы атак на рой

### 1. Sybil-атака
Атакующий создаёт множество фальшивых идентичностей, чтобы:
- Перетянуть на себя лидерство
- Скомпрометировать консенсус
- Получить больше ресурсов

**Защита:**
- Регистрация идентичностей (PKI)
- Проверка ресурсов (вычислительная мощность, память)
- Proof of work, proof of stake
- BFT (глава 6) частично защищает

### 2. Eclipse-атака
Атакующий изолирует агента от остального роя, окружая его фальшивыми соседями.

**Защита:**
- Проверка идентичности соседей
- Множественные каналы связи
- Random peer selection

### 3. Jamming
Подавление радиосвязи.

**Защита:**
- Frequency hopping (FHSS)
- Spread spectrum
- Резервные каналы (mesh с переключением)
- Оптическая связь (лазеры, фонари)

### 4. GPS-spoofing
Подмена GPS-сигнала, чтобы дрон думал, что он в другом месте.

**Защита:**
- INS (Inertial Navigation System) как backup
- Multi-GNSS (GPS + ГЛОНАСС + Galileo + BeiDou)
- Cross-check с лидаром и камерой
- Anti-spoofing приёмники

### 5. Подмена команд
Атакующий посылает фальшивые команды от имени лидера.

**Защита:**
- Криптографические подписи команд
- Sequence numbers (защита от replay)
- Multi-factor (команда + подтверждение от нескольких)

### 6. Replay-атака
Атакующий записывает легитимные команды и воспроизводит их позже.

**Защита:**
- Timestamps
- Nonces (одноразовые числа)
- Sequence numbers

### 7. Атаки на ML-модели
- **Adversarial examples** — специально созданные входы, обманывающие нейросеть
- **Data poisoning** — отравление обучающих данных
- **Model inversion** — восстановление обучающих данных из модели
- **Backdoor** — скрытый триггер в модели

**Защита:**
- Adversarial training
- Differential privacy
- Federated learning с проверкой
- Anomaly detection

### 8. Атаки на LLM-агентов
- **Prompt injection** — вредоносные инструкции в данных
- **Jailbreak** — обход safety-ограничений
- **Tool misuse** — агент вызывает неправильный API
- **Context overflow** — забивание контекста
- **Confused deputy** — агент с полномочиями выполняет вредоносный запрос

**Защита:**
- Sandboxing
- Структурированные выходы (JSON schema)
- Allow-list инструментов
- Human-in-the-loop
- Constitutional AI
- Watermarking ответов

### 9. Каскадные атаки
Один скомпрометированный агент отравляет других через mesh.

**Защита:**
- Reputation systems
- BFT-консенсус
- Периодическая ре-аутентификация

## Архитектура безопасного роя

### Defense in depth

```
Уровень 1: Физический
  - Защита оборудования (tamper-evident)
  - Геофенсинг
  - Физические барьеры
Уровень 2: Связь
  - Шифрование (AES-256, ChaCha20)
  - Аутентификация (mutual TLS, certificates)
  - Frequency hopping
Уровень 3: Сеть
  - Mesh-протоколы с проверкой (RPL с security)
  - DDoS-защита
  - Intrusion Detection
Уровень 4: Консенсус
  - BFT (если нужна byzantine-устойчивость)
  - Подписи всех сообщений
  - Кворум
Уровень 5: Приложение
  - Sandboxing
  - Валидация входов/выходов
  - Allow-list операций
Уровень 6: LLM (если применимо)
  - Constitutional AI
  - Constitutional classifiers
  - Rate limiting
  - Human-in-the-loop
Уровень 7: Мониторинг
  - Anomaly detection
  - Трассировка
  - Аудит
```

### Zero Trust Architecture

Принцип: **никому не доверяй, всё проверяй**.

- Каждый агент аутентифицируется при каждом взаимодействии
- Каждое сообщение подписывается
- Каждое действие логируется
- Права выдаются по принципу минимальных привилегий

**Примеры реализации:**
- Istio (service mesh для Kubernetes)
- SPIFFE/SPIRE (идентичности для workload)
- HashiCorp Vault (секреты)
- OPA (Open Policy Agent) — политики

### Цифровые подписи и PKI

Каждый агент имеет:
- **Приватный ключ** — для подписи (никогда не покидает устройство)
- **Сертификат** — выдаётся удостоверяющим центром (CA)
- **Цепочку доверия** — корневой CA → промежуточный → агент

**Проблемы:**
- Что если CA скомпрометирован? (Решение: HSM, multi-sig)
- Что если агент украден? (Решение: revocation list)
- Что если приватный ключ украден? (Решение: secure element, attestation)

### Hardware Root of Trust

Аппаратные компоненты, которым можно доверять:
- **TPM (Trusted Platform Module)** — криптографический чип
- **Secure Enclave** (Apple) — изолированный процессор
- **TrustZone** (ARM) — изолированная среда
- **SE (Secure Element)** — в SIM-картах, платёжных картах

Используется для:
- Хранения ключей
- Secure boot
- Attestation (доказательство, что код не изменён)
- Изолированного исполнения критических операций

### Secure Boot и Attestation

**Secure boot:**
- При включении устройства проверяется подпись bootloader'а
- Затем — подпись ОС
- Затем — подпись приложений
- Если что-то не совпадает — устройство не загружается

**Remote attestation:**
- Устройство доказывает удалённому серверу, что на нём легитимный код
- Используется в IoT для проверки прошивки

### Intrusion Detection Systems (IDS) для роя

**Сетевые IDS:**
- Snort, Suricata — для mesh-сети
- Обнаружение аномального трафика
- Паттерны атак

**Behavioral IDS:**
- ML-модель нормального поведения агента
- Аномалии = тревога
- Проблема: false positives

**Distributed IDS:**
- Каждый агент мониторит соседей
- Голосование по подозрительному поведению
- BFT-решение об исключении

## Безопасность по типу роя

### Воздушные дроны
- **Главные риски:** GPS-spoofing, перехват управления, потеря связи
- **Регуляторика:** FAA/EASA требует Remote ID (с 2023 в США)
- **Стандарты:** ASTM F3322-22, ISO 21384-3
- **Специфика:** китайские DJI под подозрением в западных военных

### Наземные роботы
- **Главные риски:** физический перехват, tampering
- **Решения:** tamper-evident корпус, alarm при открытии
- **Видеонаблюдение** — отдельная проблема privacy

### IoT-рой
- **Главные риски:** Mirai-style botnet, утечка данных
- **Решения:** уникальные пароли, обновления прошивки, сегментация сети
- **Стандарты:** NIST IR 8259, ETSI EN 303 645

### Программные агенты (LLM)
- **Главные риски:** prompt injection, excessive agency
- **Решения:** sandboxing, валидация выходов, allow-list tools
- **Стандарты:** OWASP Top 10 for LLMs (2023+)

## Приватность

### Видео с дронов
- Дроны снимают всё подряд
- Законодательство (GDPR, ФЗ-152) требует защиты
- Решения: on-device processing (обработка на борту), blurring лиц, edge-аналитика

### LLM-агенты и приватность
- LLM могут «запомнить» данные из контекста
- LLM могут выдать приватную информацию при неправильном запросе
- Решения: локальные LLM, differential privacy, фильтрация выходов

### Данные с IoT
- Датчики собирают всё (включая то, что не должны)
- Решения: data minimization, encryption at rest, TTL на данные

- **FAA Part 107** (США) — коммерческие дроны
- **EU 2019/947** (ЕС) — единые правила

- **Geo-fencing** — запретные зоны

- **GDPR** (ЕС) — защита персональных данных
- **California IoT Security Law** (SB-327) — уникальные пароли
- **UK PSTI Act** (2024) — запрет дефолтных паролей
- **NIS2 Directive** (ЕС) — кибербезопасность критических инфраструктур

### LLM-агенты
- **EU AI Act** (принят 2024) — классификация по рискам
- **NIST AI Risk Management Framework** (США)
- **OWASP Top 10 for LLM Applications**
