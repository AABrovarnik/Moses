# Глава 8. Эмерджентное поведение и предсказуемость

## Что такое эмерджентное поведение

**Эмерджентность** (emergence) — это когда глобальное поведение системы не очевидно из локальных правил её частей.

Примеры из роевых систем:
- **Стая птиц** — из трёх правил (держать дистанцию, согласовать скорость, тянуться к центру) рождаются волны, формации, синхронные повороты
- **Муравейник** — из простых правил каждого муравья возникает архитектура, вентиляция, ферма для тли
- **Бот-рой на складе** — из локальных правил объезда рождается глобально оптимальный трафик
- **Трафик на дороге** — из правил отдельных водителей рождаются пробки (иногда)

Эмерджентное поведение — это **и благо, и проклятие**:
- Благо: не нужно проектировать глобальное поведение, оно возникает само
- Проклятие: его трудно предсказать и отладить

## Три уровня предсказуемости

### Level 0: полностью детерминированный
- Каждый агент выполняет предписанный сценарий
- Глобальное поведение — сумма поведений
- Пример: конвейер на заводе
- **Не рой** — это автоматизация

### Level 1: локально детерминированный
- Каждый агент выполняет детерминированные правила, но **реагирует на соседей**
- Глобальное поведение **возникает** из локальных правил
- Пример: Boids, ACO
- **Это рой в строгом смысле**

### Level 2: локально стохастический
- Агент использует случайность в решениях
- Глобальное поведение — статистическое
- Пример: муравьиный феромон
- **Это рой с элементами исследования**

### Level 3: обучаемый
- Агент меняет свои правила на основе опыта
- Глобальное поведение — обученное
- Пример: нейросетевые рои, RL-агенты
- **Самый современный, наименее предсказуемый**

## Проблемы предсказуемости

### Модель Бёйенса

Систему можно описать на четырёх уровнях:
- **P** — физическая реализация (код, железо)
- **I** — интерпретация модели (как мы понимаем поведение)
- **M** — модель (формальное описание)
- **S** — спецификация (что система **должна** делать)

Предсказуемость требует, чтобы все 4 уровня были согласованы. В рое это сложно.

### Когда предсказуемость нарушается

- **Эмерджентность:** локальные правила ⇒ глобальное поведение, которое никто не проектировал
- **Каскадные эффекты:** малые изменения в одном агенте ⇒ большие в системе (бабочка Лоренца)
- **Feedback loops:** агент A влияет на B, B — на A, A усиливает, B усиливает...
- **Phase transitions:** при 50% работают, при 49% — нет (порог)
- **Галлюцинации LLM:** в программных роях (глава 4) — самый непредсказуемый элемент

## Как проектировать предсказуемые рои

### Принцип 1: явные инварианты

Задать свойства, которые **должны** сохраняться независимо от поведения:
- **Safety invariant:** «робот никогда не приближается к человеку ближе 1 м»
- **Liveness invariant:** «каждая задача будет выполнена за 2 часа»
- **Coordination invariant:** «не более 3 роботов в одной зоне»
- **Resource invariant:** «батарея не разрядится до возврата на базу»

Инварианты проверяются формально (model checking) или эмпирически (тесты).

### Принцип 2: ограниченная автономия

Чем уже рамки, тем предсказуемее:
- **Reactive agents** — только стимул-ответ, нет памяти ⇒ очень предсказуемы
- **Deliberative agents** — план, исполнение, пересмотр ⇒ менее предсказуемы
- **Learning agents** — меняют поведение ⇒ наименее предсказуемы

На практике: для критических систем (военные, медицинские) — реактивные. Для исследовательских (научный рой) — обучаемые.

### Принцип 3: наблюдаемость

Чтобы роем можно было управлять, нужно его **видеть**:
- **Логирование** всех решений каждого агента
- **Телеметрия** состояния (батарея, позиция, задача)
- **Трассировка** событий
- **Dashboard** в реальном времени

Примеры:
- **ROS** — стандарт для наземных роботов, логирование встроено
- **OpenTelemetry** — для программных агентов
- **Prometheus + Grafana** — стандарт для observability

### Принцип 4: тестируемость

Рой нужно тестировать так же, как обычный код:
- **Unit tests** — отдельные правила агента
- **Integration tests** — взаимодействие 2–3 агентов
- **Swarm tests** — 10–1000 агентов в симуляции
- **Field tests** — реальные испытания
- **Chaos engineering** — намеренно ломать агентов и смотреть, что будет

Инструменты:
- **ARGoS** — симулятор swarm robotics
- **Webots** — общий симулятор роботов
- **Gazebo** — стандарт для ROS
- **Mesa** — агент-базированное моделирование (Python)
- **NetLogo** — классика для MAS-симуляций
- **LLM-симуляции** — AgentVerse, ChatDev

### Принцип 5: формальная верификация

Для критических систем:
- **Model checking** — проверка, что модель удовлетворяет спецификации (P, M, S, I)
- **Theorem proving** — Coq, Isabelle
- **TLA+** — для распределённых алгоритмов
- **Stateless model checking** — для многоагентных систем

Примеры:
- **Amazon TLA+** — проверка DynamoDB, S3
- **Microsoft TLA+** — проверка Cosmos DB
- **DARPA HACMS** — формальная верификация для военных дронов (встраиваемая ОС seL4)

### Принцип 6: human-in-the-loop

Человек в контуре управления — самый надёжный способ предсказуемости:
- **Teleoperation** — прямое управление
- **Supervisory control** — человек задаёт цели, рой исполняет
- **Approval gates** — критические действия требуют подтверждения
- **Override** — человек может в любой момент взять управление

## Когда предсказуемость не нужна

- **Исследовательские задачи** — мы хотим неожиданных решений
- **Адаптация к неизвестной среде** — Марс, океанское дно, эпидемия
- **Творческие задачи** — генерация идей, дизайн
- **Кибербезопасность red team** — атакующий рой должен быть непредсказуемым

## Управление эмерджентностью

### Сдерживание (Containment)
Ограничить пространство, в котором может проявиться эмерджентное поведение:
- Геофенсинг (физические или виртуальные границы)
- Time bounds (максимальное время на задачу)
- Resource bounds (максимальное использование ресурсов)

### Усиление (Amplification)
Если нужно — **стимулировать** эмерджентное поведение:
- Изменить параметры (пороги, веса)
- Ввести/убрать возмущения
- Добавить положительную обратную связь

### Наблюдение (Monitoring)
Отслеживать признаки эмерджентного поведения:
- **Метрики:** разнообразие поведений, синхронность, плотность
- **Аномалии:** внезапные изменения, стадный эффект
- **Сходимость:** агенты сходятся к одному решению (хорошо или плохо?)

## Кейсы эмерджентного поведения

### Полезное

- **Boids-волны:** в стае птиц возникают «волны» движения — никто не проектировал, но это работает
- **Ant trails:** муравьи находят кратчайший путь — никто не планировал
- **Distributed consensus:** агенты соглашаются с состоянием без центрального голосования
- **Колония роботов формирует структуру:** рой ANYmal построил башню в ETH Zürich (2014) — из локальных правил

### Вредное

- **Deadlock:** роботы заблокировали друг друга, никто не может двинуться
- **Oscillation:** агенты переключаются между двумя состояниями бесконечно
- **Stampede:** все идут в одну сторону, перегрузка
- **Gossip amplification:** gossip-протокол распространяет ложную информацию быстрее, чем ожидалось
- **LLM agents in feedback loop:** Агент A отвечает агенту B, B отвечает A, и так 100 раз — контекст «взрывается»

### Нейтральное, но неожиданное

- **Эмерджентные роли:** в рое 100 одинаковых агентов некоторые начинают играть разные роли (лидер, разведчик, охранник) — без явного назначения
- **Фазовые переходы:** при 10 агентах система ведёт себя одним образом, при 11 — совсем другим
- **Краевые эффекты:** поведение на границе роя (10% на краю) отличается от центра (90% внутри)

## Формальные методы для роёв

### Linear temporal logic (LTL)
Спецификация поведения на языке логики:
- **GF p** — «бесконечно часто p»
- **FG p** — «в конце концов всегда p»
- **G(p → Xq)** — «если p, то на следующем шаге q»

Применяется для:
- Specification of swarm behavior
- Verification
- Synthesis of controllers (SLAP — Synthesis from LTL with Action-based Planning)

### Probabilistic model checking
Учёт вероятностного поведения:
- PRISM, Storm
- Для анализа LLM-роёв (вероятность галлюцинации)

### Multi-agent path finding (MAPF)
Формальная задача:
- N агентов, граф, начальные и конечные позиции
- Найти пути, чтобы агенты не столкнулись
- Алгоритмы: CBS, ECBS, SAT-based

### Runtime verification
Проверка в реальном времени:
- Монитор следит за трассой выполнения
- Если обнаружено нарушение — сигнал тревоги
- Пример: COPILOT, JavaMOP

## Грань между роем и хаосом

Эмерджентное поведение — палка о двух концах. Слишком мало — система жёсткая, не адаптивная. Слишком много — система непредсказуема, опасна.

**Золотая середина:**
- Локальные правила **достаточно просты** для анализа
- Глобальное поведение **достаточно богато** для адаптации
- Инварианты **проверяемы** формально
- Человек **в контуре** для критических решений
