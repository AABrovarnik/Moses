# Глава 13. Дизайн роя: практическое руководство

## Зачем эта глава

Главы 1–12 разбирают, **что** такое рой, **как** он работает, и **почему** важна этика. Эта глава — про **как именно его проектировать**. Целевая аудитория — инженер, впервые столкнувшийся с задачей «нужен рой агентов для X».

## 1. Перед стартом: 7 вопросов, на которые нужно ответить

Прежде чем писать код, нужно ответить на семь вопросов. Если не получается — значит, задача плохо поставлена.

### 1.1 Какую задачу решаем?
Не «хочу попробовать PSO», а **конкретный результат**, который должен получиться. Например:
- «Доставить 1000 посылок в городе за 8 часов с вероятностью успеха 99%».
- «Обнаружить источник задымления в здании 100×100 м за 5 минут».
- «Обойти склад 1000 стеллажей, построив карту с точностью 5 см».

### 1.2 Сколько агентов нужно?
Не «чем больше, тем лучше», а **минимально необходимая численность + запас**. Принцип: начните с минимума, наращивайте до тех пор, пока метрика растёт.

Типичные диапазоны:
- 5–20 агентов: SAR на малом объекте, обход одного этажа.
- 50–200: складская логистика, доставка в микрорайоне.
- 200–1000: городское патрулирование, агрооперации.
- 1000–100 000: военный рой, мониторинг леса, edge-облачные вычисления.

### 1.3 Сколько агентов может быть потеряно?
- **0% потерь**: медицинские, детские, военные миссии.
- **10% потерь**: SAR, агро, военные.
- **50% потерь**: разведка в опасной среде.
- **100% потерь**: одноразовые барражирующие боеприпасы.

От этого зависят: выбор железа (дешёвое vs. защищённое), плотность формации, степень автономности.

### 1.4 Где проходит граница автономности?
- **Полностью ручной режим**: рой — это дистанционно управляемые агенты.
- **Teleoperation + assist**: человек ведёт, рой помогает.
- **Полуавтономный**: рой сам планирует, человек подтверждает ключевые решения.
- **Полностью автономный**: рой сам, человек наблюдает и вмешивается при исключениях.
- **Сверх-автономный**: рой может игнорировать человека, если считает его решение ошибочным (спорно, обсуждается).

### 1.5 Какая среда?
- **Закрытая**: склад, цех, лаборатория.
- **Полуоткрытая**: кампус, парк, складской двор.
- **Открытая**: город, поле, лес.
- **Агрессивная**: зона ЧС, военный театр, под водой, в космосе.

Каждая среда диктует свои требования к сенсорам, связи, защите.

### 1.6 Какой бюджет и сроки?
- MVP за 1 месяц: 3 агента, открытый стек, симуляция.
- Продукт за 6 месяцев: 10–30 агентов, продакшн-стек, пилотирование.
- Промышленное решение за 18–36 месяцев: 100+ агентов, сертификация, развёртывание.

### 1.7 Какие метрики успеха?
Без метрик невозможно сказать, что рой работает. Минимум:
- **Mission success rate** (% выполненных задач).
- **Time-to-completion** (среднее, медиана, 95-й перцентиль).
- **Cost per mission** (полный TCO с учётом износа).
- **Resilience** (как рой ведёт себя при потере 25/50/75% агентов).
- **Safety incidents** (количество и тяжесть).

## 2. Выбор архитектуры: дерево решений

```
                    ┌─ Задача требует глобального оптимума?
                    │  (например, минимизация суммарного пути)
                    │
                    ├─ Да → Централизованная или иерархическая
                    │       с одним мастером и алгоритмом оптимизации
                    │
                    └─ Нет → Задача терпит локальный оптимум?
                              │
                              ├─ Нет → Иерархическая с кластерами
                              │        (компромисс)
                              │
                              └─ Да → Децентрализованная
                                      (максимальная отказоустойчивость)
```

### 2.1 Матрица «архитектура × задача»

| Задача | Рекомендуемая архитектура | Почему |
|---|---|---|
| Складская логистика | Централизованная | Глобальный план маршрутов, минимум коллизий |
| SAR | Децентрализованная | Зона может быть опасной, связь нестабильна |
| Агро (мониторинг полей) | Децентрализованная | Покрытие, нет нужды в глобальном оптимуме |
| Доставка в городе | Иерархическая | Кластеры по районам, центральный диспетчер |
| Военный рой разведки | Децентрализованная | Связь подавляется, потери неизбежны |
| Управление БПЛА-световое шоу | Централизованная | Нужна точная синхронизация |
| Управление дата-центром | Иерархическая | Менеджеры серверов, координатор кластера |
| Программный LLM-рой | Иерархическая | Супервайзер + специализированные агенты |

## 3. Выбор алгоритма: библия алгоритмов роя

### 3.1 Когда какой алгоритм

| Задача | Алгоритм | Почему |
|---|---|---|
| Глобальная оптимизация | PSO | Прост, хорошо параллелится, хорош для непрерывных пространств |
| Комбинаторная оптимизация (маршруты) | ACO, GA | ACO хорош для графов, GA универсален |
| Достижение консенсуса | BFT, Raft, Paxos | Формальные гарантии |
| Покрытие территории | Потенциальные поля, виртуальные силы | Локальные правила дают глобальный эффект |
| Формирование строя | Reynolds boids, leader-follower | Биологически инспирированно, интуитивно |
| Распределённое SLAM | ORB-SLAM3 + многороботный вариант | Зрелая технология |
| Распределённое обучение | FedAvg, FedProx | Устоявшийся стандарт |
| Координация LLM-агентов | CrewAI, LangGraph, AutoGen | См. главу 15 |

### 3.2 Гибридный подход

Часто лучший результат даёт **гибрид**:
- ACO для верхнего уровня (планирование маршрутов),
- PSO для нижнего (управление формацией),
- BFT для критических решений (выбор лидера),
- ML-компонент для адаптации к среде.

Пример: рой дронов для доставки. Верхний уровень — ACO планирует маршруты по графу улиц. Средний — каждый дрон летит по маршруту, избегая препятствия потенциальными полями. Нижний — управление моторов PID-регулятором. Плюс BFT-консенсус при конфликте в зоне.

## 4. Выбор симулятора

Прежде чем запускать в железе — месяцы симуляции. Симуляторы:

### 4.1 Общего назначения

- **Gazebo (Ignition)** — стандарт для ROS, хорош для роботов-манипуляторов и наземных.
- **Unity / Unreal Engine** — фотореалистичная графика, дроны, дроны в помещении.
- **AirSim (Microsoft)** — дроны и автомобили, высокое качество сенсоров.
- **CARLA** — автомобили, город, погодные условия.

### 4.2 Специализированные для роёв

- **ARGoS** — крупные рои (до 10 000+), низкие требования к железу, идеален для экспериментов.
- **NetLogo** — простой агент-ориентированный, для обучения и прототипов.
- **MESA (Python)** — агент-ориентированный, гибкий, интеграция с Jupyter.
- **SwarmSim (C++/ROS)** — для ROS-стека.
- **Stage / Stage ROS** — лёгкий 2D-симулятор.
- **Webots** — 3D, кросс-индустриальный, хорош для манипуляторов.

### 4.3 Критерии выбора

- **Масштаб**: NetLogo — до 1000, ARGoS — до 100 000, Gazebo — до 100.
- **Реализм сенсоров**: Gazebo/AirSim/Webots — высокий, NetLogo/ARGoS — низкий.
- **Скорость итерации**: NetLogo/ARGoS — секунды, Gazebo — минуты.
- **Лицензия**: открытые vs. проприетарные.
- **Поддержка ROS**: Gazebo, Webots, AirSim — да, NetLogo — нет.
- **Headless режим** (для CI/CD): ARGoS, MESA, Gazebo.

### 4.4 Пайплайн разработки

```
1. Сценарий в NetLogo (быстрая проверка гипотезы)
   ↓ работает?
2. Перенос в ARGoS или MESA (проверка на 100+ агентах)
   ↓ работает?
3. Перенос в Gazebo/AirSim (проверка с реалистичными сенсорами)
   ↓ работает?
4. Hardware-in-the-loop (HIL) — реальный агент + симулированные соседи
   ↓ работает?
5. Polysim — несколько реальных агентов + симулированные
   ↓ работает?
6. Малое реальное развёртывание (3–5 агентов)
   ↓ работает?
7. Полное развёртывание
```

## 5. Метрики успеха: что измерять

### 5.1 Производительность

- **Throughput** (задач/час)
- **Latency P50/P95/P99** (время отклика)
- **Mission success rate** (% выполненных)
- **Coverage rate** (доля покрытой территории за время T)
- **Convergence time** (для оптимизационных задач)

### 5.2 Отказоустойчивость

- **Degradation curve** (как падает метрика при потере 25/50/75% агентов)
- **Recovery time** (после восстановления связи / замены агента)
- **Critical mass** (минимальное число агентов для выполнения задачи)
- **Byzantine tolerance** (сколько злонамеренных/сломанных агентов рой выдерживает)

### 5.3 Безопасность

- **Collision rate** (столкновения на 1000 часов)
- **Near-miss rate** (опасные сближения)
- **Boundary violation rate** (выходы за геозоны)
- **Unauthorized action rate** (действия вне мандата)

### 5.4 Экономика

- **Cost per mission**
- **TCO per year**
- **Energy per task** (Вт·ч / задача)
- **Agent utilization** (% времени агент в полезной работе)

### 5.5 Этика (см. главу 12)

- **Bias score** (по подгруппам)
- **Explainability score** (% решений с понятным объяснением)
- **Operator override rate** (как часто человек вмешивается)

## 6. A/B-тестирование роёв

### 6.1 Зачем

Чтобы выбрать между двумя алгоритмами, нужны **честные эксперименты** на одинаковых сценариях.

### 6.2 Метод

1. **Зафиксировать сценарии**: 10–100 типовых задач с разной сложностью.
2. **Запустить оба варианта** на каждом сценарии N раз (Monte Carlo).
3. **Собрать метрики**.
4. **Статистический тест**: t-тест, Манн-Уитни для непараметрических, ANOVA для многих групп.
5. **Эффект и доверительный интервал** — не только p-value, но и размер эффекта.

### 6.3 Контроль переменных

Менять **одну** переменную за раз:
- алгоритм,
- число агентов,
- топология связи,
- параметры среды,
- параметры агента (скорость, дальность сенсоров).

### 6.4 Документирование

Каждый эксперимент — в лог:
- commit hash,
- параметры конфигурации,
- сценарий,
- результаты,
- графики.

Без этого A/B-тестирование превращается в «мне кажется, новый алгоритм лучше».

## 7. Red team для роя

### 7.1 Зачем

Прежде чем злоумышленник найдёт слабость — её должна найти **ваша** команда.

### 7.2 Что ищут

- **Logic bugs**: как заставить рой зациклиться, рассыпать формацию, перегрузить канал.
- **Состязательные атаки**: как обмануть сенсоры (стикеры на знаки, ИК-помехи).
- **Этические дилеммы**: как заставить рой принять неэтичное решение.
- **Failure mode**: что произойдёт при отказе лидера, при разрыве связи, при потере 90% агентов.
- **Side effects**: что произойдёт с **другими** системами, если рой начнёт вести себя иначе.

### 7.3 Метод

1. **Adversarial scenario design**: придумать 50 сценариев, в которых рой должен сломаться.
2. **Fuzzing**: подавать на вход случайные/грамматически-неправильные данные.
3. **Chaos engineering**: отключать компоненты (Chaos Monkey для роя).
4. **Penetration testing**: имитация атаки на коммуникации.
5. **Stress test**: перегрузка (1000 команд в секунду на агента, 0% потерь пакетов, экстремальные температуры).

### 7.4 Инструменты

- **ARGoS Chaos** — модуль для внесения неисправностей.
- **ROS Chaos** — Chaos Engineering для ROS-стека.
- **Custom fuzzer** — простые Python-скрипты для подачи мусорных данных.

## 9. Типичные ошибки

### 9.1 Инженерные

| Ошибка | Последствие | Как избежать |
|---|---|---|
| Жадное масштабирование | Рой из 1000 агентов валится, потому что на 100 не тестировали | Тестировать на 1x, 3x, 10x |
| Игнорирование задержки связи | Агенты действуют на основе устаревших данных | Моделировать задержку явно |
| Одинаковые агенты | Один баг валит всех | Гетерогенность, разные версии прошивки |
| Доверие к симулятору | В железе всё иначе | Hardware-in-the-loop, пилот |
| Отсутствие логирования | При инциденте нечего анализировать | Полное логирование решений и состояний |
| Нет graceful degradation | Потеря 10% агентов = отказ | Тест на деградацию |
| Один язык / один фреймворк | Vendor lock-in | Гетерогенный стек |

### 9.2 Архитектурные

| Ошибка | Последствие |
|---|---|
| Слишком умные агенты | Поведение непредсказуемо, невозможно отлаживать |
| Слишком тупые агенты | Не справляются с динамикой среды |
| Глобальная координация там, где не нужна | Лишний расход связи и времени |
| Локальная координация там, где нужна глобальная | Субоптимальные решения |
| Слишком много уровней иерархии | Задержки принятия решений |
| Слишком мало уровней | Потеря масштабируемости |

### 9.3 Организационные

| Ошибка | Последствие |
|---|---|
| «Сделаем MVP и посмотрим» | MVP — это не рой, это игрушка |
| Отсутствие этического паспорта | Запуск блокируется регулятором |
| Нет владельца продукта | Рой развивается хаотично |
| Несоответствие регуляторике | Штрафы, отзыв, репутационный ущерб |
| Общественное сопротивление | Проект блокируется на старте |

## 10. Шаблоны проектирования

### 10.1 Экспертная панель

```
Координатор
  ├─ Агент-эксперт A (предлагает решение)
  ├─ Агент-эксперт B (предлагает решение)
  ├─ Агент-эксперт C (предлагает решение)
  └─ Агент-критик (оценивает решения A, B, C)
```

Применяется в LLM-роях для задач с разными точками зрения.

### 10.2 Map-Reduce для роя

```
Агенты-сборщики (каждый обрабатывает свой кусок)
  ↓
Агенты-агрегаторы (сводят результаты по зонам)
  ↓
Координатор (общее решение)
```

Применяется в задачах типа «обойти территорию и построить карту».

### 10.3 Супервайзер с подчинёнными

```
Супервайзер (получает задачу, декомпозирует)
  ├─ Worker 1 (выполняет подзадачу 1)
  ├─ Worker 2 (выполняет подзадачу 2)
  └─ Worker N (выполняет подзадачу N)
```

Применяется в чётко декомпозируемых задачах.

### 10.4 Теневой рой (для тестирования)

Реальный рой + параллельный симулированный рой, получающий те же входы. Используется для:
- **what-if анализа** (что было бы, если бы агент поступил иначе),
- **обучения** (реальный рой учится на примере симулированного),
- **предсказания** (куда пойдёт рой дальше).

### 10.5 Иерархия внимания

```
Глобальный контекст (обновляется раз в минуту)
  ↓
Региональный контекст (обновляется раз в 10 секунд)
  ↓
Локальный контекст (обновляется 100 раз в секунду)
```

Позволяет агенту не тратить ресурсы на то, что редко меняется.

## 11. Инфраструктура

### 11.1 Связь

| Среда | Технология | Дальность | Задержка |
|---|---|---|---|
| Внутри помещения | Wi-Fi 6/6E, UWB | 30–100 м | 1–10 мс |
| Город | 4G/5G, LoRaWAN | 1–10 км | 20–100 мс |
| Открытая местность | Mesh-радио, satellite | 10–1000 км | 50 мс — 1 с |
| Под водой | Акустика | 1–10 км | 1–10 с |
| В космосе | Оптическая, радио | межпланетные | секунды-минуты |

### 11.2 Энергия

- **Li-ion батареи**: 150–250 Вт·ч/кг, для большинства задач.
- **Li-Po**: 130–200 Вт·ч/кг, для дронов.
- **Li-S, Li-air**: 400–1000 Вт·ч/кг, в разработке.
- **Топливные элементы (водород)**: 500–1000 Вт·ч/кг, для долгих миссий.
- **Солнечные панели**: 100–300 Вт/м², для стационарных.
- **Беспроводная зарядка**: для постоянно дежурящих роёв (склад).

### 11.3 Зарядка / обслуживание

- **Автоматические зарядные станции** (дроны прилетают, заряжаются).
- **Сменные батареи** (робот подъезжает, рука/человек меняет).
- **Полная замена** (для дешёвых одноразовых агентов).

### 11.4 Edge-облако

- **On-board** (только локально): низкая задержка, ограниченная мощность.
- **Edge** (близкий сервер): средняя задержка, средняя мощность.
- **Cloud** (удалённый): высокая задержка, высокая мощность.
- **Гибрид**: критические решения — on-board, аналитика — cloud.

## 13. Жизненный цикл

```
Концепция → MVP → Пилот → Продукт → Развитие → Закат
   │          │        │        │          │         │
   └──────────┴────────┴────────┴──────────┴─────────┘
       каждый этап имеет свои метрики и gate-критерии
```

### 13.1 Концепция
- Сформулированы 7 вопросов.
- Проведена оценка технической и этической реализуемости.
- Решение: идём / не идём.

### 13.2 MVP
- 3–5 агентов в симуляции.
- 1 алгоритм, 1 сценарий.
- Метрика успеха подтверждена.

### 13.3 Пилот
- 10–30 агентов в реальности.
- 2–3 сценария.
- Сбор данных для улучшения.

### 13.4 Продукт
- 100+ агентов.
- Сертификация, лицензирование.
- Этический паспорт.

### 13.5 Развитие
- Расширение сценариев.
- Новые алгоритмы (A/B-тест).
- Масштабирование.

### 13.6 Закат
- Плановый: замена на новую версию.
- Аварийный: инцидент → отзыв → расследование.

## 15. Итог

Дизайн роя — это **итеративный процесс**, и первая итерация почти наверняка будет ошибочной. Главное — быстро проверять гипотезы в симуляции, на малом масштабе, с метриками. Этика, безопасность и регуляторика должны быть встроены с самого начала, а не приклеены в конце.

Хороший дизайн роя — это **баланс** между:
- автономностью и контролем,
- эффективностью и безопасностью,
- масштабом и сложностью,
- инновацией и ответственностью.

Баланс этот для каждой задачи свой, и универсального рецепта нет. Но чек-листы, метрики и процессы — общие.
