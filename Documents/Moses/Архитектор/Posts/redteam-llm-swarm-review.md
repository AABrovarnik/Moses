# Red-team review: программный рой 3–5 LLM-агентов (с выбором моделей)

> **Исходник:** `../Промпт.md` — задача red-team review архитектуры отказоустойчивой системы AI-агентов (Venya Core + Izya-Speedy/Deep/Max + Moysha++ + Watchman).
> **Цель этого отчёта:** проанализировать архитектуру через призму реальной реализации **программного роя из 3–5 LLM-агентов** с учётом **бюджетных ограничений** и профиля потерь **до 30% агентов в неделю на 2–3 дня с последующим восстановлением**, и дать рекомендации по технической реализации и набору LLM.
> **Дата:** 2026-06-12.

| Параметр | Решение |
|---|---|
| **Вердикт** | Принять с изменениями |
| **Целевой размер роя** | **3 агента** (1 быстрый, 1 сильный, 1 критик-резерв) + 1 «HOLD-only»-агент Venya Core |
| **Бюджет** | **$50–120/мес** |

## 1. Вердикт

**Жизнеспособна, но перегружена.** 12 принципов исходного промпта — это архитектура **mission-critical** уровня космической группировки, а не бюджетный рой из 3–5 агентов. Избыточная сложность **удвоит** операционные расходы и **утроит** вероятность багов, которые сами станут источником отказов. Принципы «чем слабее преемник, тем уже полномочия» и «один инцидент — одно уведомление» — зрелые и рабочие, их нужно оставить. Пункты про 4 типа агентов, 5 каналов и 3 пакета передачи требуют радикального упрощения под заданный размер роя.

## 2. Пять наиболее опасных сценариев отказа

| # | Сценарий | Почему опасен |
|---|---|---|
| 1 | **Split-brain в Venya Core** | Два экземпляра Venya Core (например, после рестарта) считают себя командирами. Без Raft/etcd с кворумом (а не «всегда один») это **неизбежно** — fencing token не решает race condition при сетевом разделении, потому что токен нельзя обновить без связи с хранилищем. |
| 2 | **Потеря контекста при передаче командования** | Новый командир получает «ролевой пакет», но **не получает** историю решений, отменённых гипотез и тупиковых веток. Через 3 передачи (что реалистично при 30% потерях в неделю) рой работает с «телеграфным» контекстом — и совершает повторные ошибки. |
| 3 | **Watchman мёртв** | Пункт 8 исходного промпта не определяет, что делает Watchdog при собственном отказе. У него нет **своего** watchdog'а. Рекурсия «кто следит за смотрящим» не решена. |
| 4 | **Деградация до HOLD-режима без возврата** | Venya Core умеет только HOLD/QUEUE/STATUS/ESCALATE. Но **кто решает, что пора вернуться к нормальной работе**, если основной провайдер ожил? Если это решает сам Venya Core — он не должен принимать это решение, т.к. «умеет только базовое». Если человек — задержка 1–2 часа на каждом восстановлении. |
| 5 | **Канальный шторм при восстановлении** | При оживании провайдера все 4 канала (Telegram × 2, WhatsApp, email) получают одно и то же уведомление о восстановлении, потому что **очередь событий не была привязана к каналу-источнику** — пункт 10 не помогает, он про дедупликацию **инцидентов**, а не **каналов**. |

## 3. Где остаются единые точки отказа (SPOF)

- **Один сервер / один VPS** — если упадёт, нет ни Venya Core, ни Watchman, ни БД.
- **Одна БД контекста** — не указан тип. SQLite локально = SPOF. Postgres в одном инстансе = SPOF. Нужна хотя бы **реплика**.
- **Один Telegram-аккаунт для Venya Command Bot** — если Telegram банит аккаунт, теряется основной канал.
- **Один DNS / домен** для webhook-эндпоинтов.
- **Один «активный провайдер» LLM в любой момент времени** — переключение требует полуавтоматической процедуры, во время которой роя **нет**.

## 4. Что избыточно сложно для первой версии

| Пункт исходного промпта | Почему избыточно | Что упростить |
|---|---|---|
| **П.7 — три пакета при передаче** | Для 3–5 агентов это 3–5 страниц ceremony | Один пакет: «что делали, где остановились, что нельзя делать» |
| **П.5 — градация полномочий по силе преемника** | Подразумевает ACL, RBAC — это месяцы работы | Бинарно: «либо может всё, либо только HOLD» |
| **П.12 — 5 каналов с автоматическим роутингом** | 5 каналов × роутинг × fallback = отдельный подсервис | 2 канала: Telegram + email. SMS — потом |
| **П.9 — «критические сводки без LLM»** | Требует отдельного шаблонизатора | Готовые шаблоны строк в Python |
| **П.4 — 4 типа агентов с разными LLM** | 4 разных API-ключа, 4 разных тарифа, 4 разных сбоя | 2 LLM (основной + критик) + 1 фолбэк |
| **П.1 — «Venya Core без обязательной зависимости от LLM»** | Оправдано — оставить | — |

## 5. Минимальная жизнеспособная версия (MVP)

```
┌─────────────────────────────────────────────────────┐
│  Watchman (cron + bash, no LLM)                     │
│  ├─ пингает Gateway каждые 60с                      │
│  ├─ проверяет cron, размер очереди, last-heard      │
│  └─ шлёт в Telegram, при SEV-1 в email             │
├─────────────────────────────────────────────────────┤
│  Venya Core (Python, 1 процесс)                     │
│  ├─ leader election через файл-лок (flock)         │
│  ├─ на старте читает БД контекста                   │
│  ├─ режимы: NORMAL | DEGRADED | HOLD                │
│  └─ эскалация в Telegram (1 канал)                  │
├─────────────────────────────────────────────────────┤
│  Агенты (3 шт., разные LLM):                        │
│  ├─ Izya-Speedy: Claude Haiku 4.5 / GPT-5-mini     │
│  │   (быстро, дёшево, обратимо)                     │
│  ├─ Izya-Deep: Claude Sonnet 4.6 / GPT-5           │
│  │   (сложно, дороже, рискованно)                   │
│  └─ Moysha++: Gemini 2.5 Pro / DeepSeek V3.2       │
│      (критик + резервный провайдер)                 │
├─────────────────────────────────────────────────────┤
│  БД контекста: SQLite (WAL) + ssh-pull реплика      │
│  на 2-й VPS раз в 5 мин                             │
├─────────────────────────────────────────────────────┤
│  Каналы: Telegram (основной) + email (SEV-1)        │
└─────────────────────────────────────────────────────┘
```

**Ключевые упрощения:**
- Командир — **один из 3 агентов** (Izya-Deep в норме, Izya-Speedy при сбое Deep, Moysha++ при сбое обоих). Никакого отдельного «Venya-LLM-агента» — он отдельный **только для HOLD**.
- **Один** пакет контекста при передаче (не три).
- **Watchman = bash-скрипт в cron**, не микросервис.
- **2 канала**, не 5.

| # | Замечание | Риск | Исправление | Приоритет |
|---|---|---|---|---|

| 10 | БД контекста — что именно? | SQLite при многопоточности ломается | SQLite + WAL, или PostgreSQL | **P2** |
| 11 | «Долговечно вне памяти LLM» — где бэкапы? | Потеря VPS = потеря истории | ssh-pull репликация раз в 5 мин | **P1** |
| 12 | Нет лимита на размер контекста | Рост БД до неприемлемого | Ротация логов, агрегация старых решений | **P3** |
| 13 | Нет тестов восстановления | Первый реальный отказ = первый ремонт | Chaos-тест раз в неделю | **P1** |

## 9. Итоговое решение

**Принять с изменениями.**

Концепция — **рабочая** и покрывает реальные классы отказов (провайдер, канал, контекст, лидер). Но текущая формулировка описывает **систему для 30–50 агентов** (а-ля Starlink ground control), а не для бюджетного роя из 3–5. **Основные изменения:**

1. Сократить LLM-провайдеров с 4 до 2 + 1 фолбэк.
2. Заменить «3 пакета при передаче» одним пакетом контекста.
3. Убрать градацию полномочий — бинарно `FULL | HOLD`.
4. Сократить каналы с 5 до 2 в MVP, добавлять по необходимости.
5. Внедрить **active-warm standby** Venya Core на 2-м VPS, не полагаться на systemd одного сервера.
6. Добавить `epoch`/`version` ко всем записям в БД.
7. Запустить MVP за 10–14 дней, не за 2–3 месяца.
8. **Еженедельные chaos-тесты** — единственный способ убедиться, что архитектура работает.

# Рекомендации по выбору LLM

## Целевые критерии

- **Бюджет**: целевой общий расход ≤ **$50–150/месяц** при 3–5 агентах.
- **Профиль потерь**: 30% в неделю = 1 агент из 3 стабильно падает → нужна устойчивость к **полному** отказу одного провайдера.
- **Качество**: основной агент — сильный reasoning, критик — независимая школа.

## Предлагаемый набор (3 LLM)

### 10.1 Izya-Speedy (быстрые, обратимые)

**Рекомендация: Claude Haiku 4.5** (или **GPT-5-mini** как альтернативу).

Это рабочая лошадка: маршрутизация задач, короткие ответы, парсинг, форматирование. **Haiku 4.5 даёт лучшее соотношение цена/качество для structured output** и в 5–10 раз дешевле Sonnet при приемлемом качестве для «не-рассуждающих» задач. GPT-5-mini — запасной вариант, чуть хуже в code, но быстрее и у некоторых лучше следование инструкциям. **При бюджете до $50/мес — Haiku 4.5**.

### 10.2 Izya-Deep (сложные, рискованные)

**Рекомендация: Claude Sonnet 4.6** (или **GPT-5** для open-ended задач).

Капитан роя. **Sonnet 4.6 — лучший для кодинга, multi-step planning, tool use; GPT-5 — лучше в open-ended рассуждениях и в написании критических сводок**. Если проект про кодинг/агенты — Sonnet 4.6, если общий разговор/анализ — GPT-5. Оба стоят $3–15 за 1M токенов, при правильном контекст-менеджменте (50–100K токенов/день) это $5–30/мес на агента.

### 10.3 Moysha++ (критик + резерв)

**Рекомендация: Gemini 2.5 Pro** (или **DeepSeek V3.2** для экономии).

Должен быть **от другого провайдера**, чем два предыдущих (Anthropic/OpenAI), чтобы выжить при их одновременном отказе. **Gemini 2.5 Pro — хорош в мультимодальности и reasoning, плюс 1M context window**. DeepSeek V3.2 — в 30–50 раз дешевле, с аналогичным качеством, но серверы в Китае (важно для privacy/санкций). **Рекомендую Gemini 2.5 Pro** — независимая школа рассуждений, стабильные API, щедрый free tier для начала.

## Почему не четыре

Четвёртый LLM добавляет:
- **+25–40%** к бюджету.
- **+50%** к поверхности отказа (новый API-ключ, новая авторизация, новая модель поведения).
- **+1 точку мониторинга** (Watchman должен следить ещё за одним).

При 3 LLM роевое поведение покрывается полностью: основной, быстрый, критик-резерв. Добавление четвёртого — это **уже специализация** (например, отдельный vision-агент), и для этого нужен отдельный use case.

## Бюджетная оценка

| Сценарий | Расход/мес | Примечание |
|---|---|---|
| Только Haiku + DeepSeek (минимальный) | **$10–30** | Достаточно для большинства задач |
| Haiku + Sonnet + Gemini (рекомендуемый) | **$50–120** | Баланс качества и цены |
| Все top-tier модели без лимитов | **$300–800** | Только если есть финансирование |

## Дополнительные меры по бюджету

1. **Жёсткий лимит на daily tokens** через API-ключ (большинство провайдеров поддерживают).
2. **Кэширование контекста** (Anthropic Prompt Caching даёт −90% стоимости повторяющихся префиксов).
3. **Local fallback** — для некритичных задач **Ollama + Mistral 7B / Llama 3.1 8B**. Бесплатно, но медленнее и хуже. Используется, когда все облачные провайдеры легли.
4. **Usage-based отключение**: если расход за день > $X, Izya-Speedy переключается на локальную модель.

## Что НЕ рекомендую

- **GPT-4 Turbo / Claude 3 Opus / Gemini 1.5 Pro** — устаревшие версии, цена та же, качество ниже.
- **Mixtral, Yi, Qwen (cloud API)** — качество ниже DeepSeek V3, цена не лучше.
- **Open-source 70B+ модели как основной агент** — для inference нужен GPU за $2000+, что не бюджетно.
- **Только один провайдер** — нарушает исходный принцип резервирования и базовую идею промпта.

**Принять набор: Claude Haiku 4.5 + Claude Sonnet 4.6 + Gemini 2.5 Pro** (или **GPT-5-mini + GPT-5 + Gemini 2.5 Pro**, если предпочитаете OpenAI). Это даёт:

## Приложение А. Минимальный код костяка (Python, псевдокод)

```python
# venya_core.py
import fcntl
import sqlite3
import time
from pathlib import Path

LOCK_FILE = "/var/run/venya_core.lock"
DB_PATH   = "/var/lib/venya/context.db"

class VenyaCore:
    def __init__(self):
        self.lock_fd = open(LOCK_FILE, "w")
        self.mode = "NORMAL"   # NORMAL | DEGRADED | HOLD

    def acquire_leadership(self) -> bool:
        try:
            fcntl.flock(self.lock_fd, fcntl.LOCK_EX | fcntl.LOCK_NB)
            return True
        except BlockingIOError:
            return False

    def heartbeat(self) -> dict:
        return {
            "ts": time.time(),
            "mode": self.mode,
            "epoch": self.read_epoch(),
            "last_decision_id": self.last_decision_id(),
        }

    def write_decision(self, agent: str, decision: dict):
        # ВСЕГДА до выполнения — write-ahead log
        with sqlite3.connect(DB_PATH) as db:
            db.execute(
                "INSERT INTO decisions VALUES (?, ?, ?, ?, ?)",
                (time.time(), agent, decision["version"],
                 decision["epoch"], str(decision))
            )

    def transfer_command(self, from_agent: str, to_agent: str, packet: dict):
        # Один пакет, а не три
        return {
            "from": from_agent,
            "to": to_agent,
            "goal": packet["goal"],
            "state": packet["state"],
            "restrictions": packet["restrictions"],
            "next_action": packet["next_action"],
            "epoch": self.read_epoch(),
            "version": self.read_version() + 1,
        }
```

```bash
  curl -s "https://api.telegram.org/bot$TG_TOKEN/sendMessage" \
    -d "chat_id=$TG_CHAT&text=$1" >/dev/null
  if [ "$2" = "SEV-1" ]; then
    echo "$1" | mail -s "[VENYA][SEV-1]" "$ALERT_EMAIL"
  fi
fi
echo "$HASH" > /tmp/venya_alert_hash
```
